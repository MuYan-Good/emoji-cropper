## 2. 技术选型决策

### 2.1 为什么选 Vite 而非 Next.js / CRA？

- **不需要 SSR**：本项目是纯前端工具，没有 SEO 需求，不需要服务端渲染
- **不需要路由**：单页应用，只有一个页面，不需要复杂路由管理
- **开发体验好**：Vite 的 HMR 速度远快于 CRA，开发效率更高
- **构建产物小**：Vite 基于 Rollup，Tree-shaking 效果好，最终产物更轻量

### 2.2 为什么选 React 而非 Vue？

- **Canvas 生态更成熟**：React 在图片处理、Canvas 操作方面的第三方库更丰富
- **社区资源充足**：遇到问题时更容易找到解决方案
- **组件复用性好**：卡片网格、预览弹窗等组件用 React 开发效率高

### 2.3 为什么选原生 Canvas API 而非 Fabric.js / Konva？

- **需求简单**：本项目只需要基础的图片裁剪和缩放，不需要复杂的图形编辑能力
- **零额外依赖**：原生 Canvas API 是浏览器内置能力，不增加包体积
- **性能最优**：没有中间层抽象，直接操作像素数据，处理速度最快
- **GIF 兼容性好**：配合 gifuct-js 可以精确控制每一帧的渲染

### 2.4 为什么选 Tailwind CSS 而非 CSS Modules / Styled Components？

- **开发速度快**：原子化 CSS 无需切换文件，在 JSX 中直接编写样式
- **一致性高**：内置设计系统（间距、颜色、圆角等），界面风格统一
- **无需命名**：避免 CSS 类名冲突和命名困难
- **响应式简单**：内置响应式前缀，桌面端适配轻松

### 2.5 为什么不需要状态管理库（Redux/Zustand）？

- **状态简单**：核心状态只有上传的图片、裁剪参数、裁剪结果列表
- **React 内置够用**：useState + useReducer + useContext 完全可以管理
- **避免过度工程**：引入状态管理库反而增加复杂度

## 3. 关键依赖说明

### 3.1 gifuct-js（GIF 处理）

GIF 动图裁剪是本项目的核心难点之一。gifuct-js 提供：
- **GIF 解析**：将 GIF 文件解析为帧序列（每帧的图像数据和延迟时间）
- **帧渲染**：将每一帧渲染到 Canvas 上进行裁剪
- **帧编码**：将裁剪后的帧序列重新编码为 GIF 文件

### 3.2 JSZip + FileSaver.js（批量下载）

- **JSZip**：在浏览器内存中创建 ZIP 文件，将裁剪后的所有图片添加进去
- **FileSaver.js**：将 ZIP 文件（或单张图片）触发浏览器下载，兼容各主流浏览器

### 3.3 react-dropzone（拖拽上传）

- 处理拖拽进入/离开/放下的事件
- 处理点击上传的 fallback
- 文件类型过滤（仅接受 PNG/GIF）
- 文件大小限制（10MB）
